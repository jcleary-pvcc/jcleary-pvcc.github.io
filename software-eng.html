Chapter 7: Software Engineering

Term Definition
Software Engineering The Process of developing, or "Engineering" a program/software.
Integrated Development Environment/td> A Software that is dedicated to developing programs, containing tools and functions to ease the process.
Association for Computing Machinery (ACM)
  • Purpose of ACM
  • Benefits of ACM membership
  • Student ACM membership
  • To further improve computing and the education thereof
  • Access to educational programs and resources, events.
  • Allowed, costs $19-62 usd, depending on option
Institute of Electrical and Electronic Engineers (IEEE)
  • Purpose of IEEE
  • Benefits of IEEE membership
  • student IEEE membership
  • Improving technology, especially electrical engineering.
  • Networking, discounts, improved education access.
  • Allowed; with various resources for careers
Software Life Cycle: Development Phase
  • Requirements analysis
  • Design
  • Implementation
  • Testing
  • Stage where required resources are researched and found
  • When the main goals/purpose of the program is fully decided
  • Where the steps outlined in the above are applied
  • Used to find errors, faults, and bugs in the program.
Modular Implementation Design principle where parts of a program can be seperated into "modules" allowing one to better manage the parts
Unified Modeling Language (UML) A markup-style language dedicated to creating diagrams to show a design process
Use Case The Specific situation a design (program, tool, etc) is meant to be used in
Use Case Diagram A Chart of a user's possible interactions with a program; often made using UML
Data Relationships:
  • One-to-One
  • One-to-Many
  • Many-to-Many
  • Data type where one record has only one corresponding record.
  • Data type where one record on end can have many on another. Commonly used for Foreign/Primary Keys.
  • Many records that can correspond to many others.
Design Pattern A pre-made design used to manage a recurring issue/use-case
Software Quality Assurance (SQA) The principle of ensuring a Software matches quality standards: design implementation, bugs, etc.
Documentation:
  • user documentation
  • system documentation
  • technical documentation
  • Information meant to inform the end user on the system
  • Documentation that is often embedded within a system, to explain the various innerworkings.
  • Documentation meant to explain what the system is, and how it works.
Intellectual property law
  • software liscense
  • software disclaimer
  • best defenses against software liability claims
  • A liscense a software owner can give to a user to allow them to use a software
  • A part of the liscense agreement/other legal documents to specify what is/isn't within the users rights
  • Ensure your programs follow design and safety principles, take good care of the program, and produce records of this care.
